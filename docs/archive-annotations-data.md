# Archive Annotations data

Archive Annotations render read-only static notes. GitHub Discussions hold the public
intake threads, but the website reads only static JSON at build time.

## where data lives

Static annotation files live in:

```text
content/annotations/<record-slug>.json
```

Each file contains a JSON array of annotations for one record only. The filename must
match every annotation's `recordSlug`.

Example:

```json
[
  {
    "id": "reader_note_001",
    "recordSlug": "why-this-exists",
    "anchorId": "p-1",
    "label": "reader note",
    "body": "This static note is sanitized source data, not a stored public submission.",
    "author": "anonymous reader",
    "created": "2026-06-15",
    "status": "approved",
    "excerpt": "This archive is not a portfolio.",
    "sourceUrl": "https://github.com/Mr-Bolan/thebolanarchives/discussions/123"
  }
]
```

The per-record GitHub Discussion registry lives in:

```text
content/annotation-discussions.json
```

That registry is generated by `npm run discussions:sync`. It maps archive record slugs to
GitHub Discussion URLs so the composer can open the relevant discussion instead of a
generic intake page. The registry records the active GitHub category; `archive-annotations`
is preferred, with `general` used as the fallback when the custom category is absent.

Allowed labels:

- `reader note`
- `field comment`
- `annotation`

Allowed statuses:

- `approved`
- `archived`

## how to add one manually

1. Pick the target record slug.
2. Add or edit `content/annotations/<record-slug>.json`.
3. Use a unique `id`.
4. Set `recordSlug` to the filename without `.json`.
5. Set `anchorId` to a paragraph or heading anchor.
6. Keep the body reviewed, sanitized, and read-only.
7. Add `sourceUrl` only when it points to the reviewed GitHub discussion for this note.
8. Run `npm run annotations:audit`.

## generated reader notes

`npm run discussions:export` can generate `reader note` entries from GitHub Discussion
comments that pass the Archive Intake Screener. Generated IDs use the
`gh_discussion_<discussion-number>_` prefix, and generated notes keep `sourceUrl` pointing
to the source GitHub comment.

Generated notes are still public repository data. Review the PR before merging.

Do not commit raw reader comments, private notes, private URLs, emails, credentials,
moderation queues, or unpublished personal context.

## anchor IDs

Paragraph anchors are generated as `p-1`, `p-2`, `p-3`, counted inside the rendered
article body after the leading record heading is removed.

Heading anchors come from the rendered `h2`, `h3`, and `h4` text. The slug rules match
`src/lib/headings.ts`: lowercase, punctuation stripped, non-alphanumeric runs collapsed
to `-`, and duplicate headings get `-2`, `-3`, and so on.

Paragraph IDs can shift when an article is edited. If paragraphs move, split, or merge,
rerun `npm run annotations:audit` and verify any `p-n` anchors still point to the right
place. For long-lived annotations, prefer a heading anchor when the note can attach to a
whole section.

## what this is not

This is not an embedded comment system. There is no custom backend, database, auth,
moderation UI, replies, likes, profiles, search, or live storage on the website. The files
are public source data, so every annotation must be safe to publish as-is.
