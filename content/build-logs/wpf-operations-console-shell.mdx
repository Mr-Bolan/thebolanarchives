# build_log_012 // wpf_operations_console_shell

## what this is

A WPF desktop shell for an operations console. It has a real application frame: .NET
Generic Host startup, dependency injection, navigation services, a fluent main window,
dashboard/data/settings pages, theme switching, icons, and view models wired through MVVM
attributes.

It is also plainly still a shell. Several screens are stubs. The dashboard is a
counter. The data page renders generated color blocks. The settings page sketches future
connection, endpoint, database, batch, machine target, and threshold-band configuration,
but the actual machine/API behavior is not implemented yet.

That makes the useful record less "what the app does" and more "what had to exist before
the app could do anything safely."

## why it exists

Some operational tools start with data or PDFs. Others start with a window.

The window matters because it decides how future work will feel: where settings live, how
navigation behaves, whether the app can grow pages without rewriting startup, how theme and
taskbar services are reached, and whether the UI is ready to hold machine configuration
without becoming a pile of event handlers.

This source is in that early frame-building stage. It is not pretending to solve the whole
workflow yet.

## the durable pieces

The source is organized around a few decisions that can survive stub screens:

- host startup uses the .NET Generic Host instead of hand-built globals
- pages and view models are registered in dependency injection
- the main window owns navigation, title bar, snackbar presenter, and fluent theme
  resources
- menu items route to dashboard, data, and settings pages
- settings state is represented as observable properties rather than direct control reads
- selected settings sections behave like a segmented mode switch
- API host/port/path and machine identity fields exist as future configuration seams

Those are not finished features. They are scaffolding. But scaffolding is still a system
choice.

## what worked

The source restored and built in an isolated clone. The build produced a Windows desktop
assembly and confirmed that the WPF/XAML wiring, package references, generated MVVM code,
and host service registration are coherent enough to compile.

The useful part is that the shell already separates responsibilities:

- startup and dependency injection live in the app host
- navigation lives in the main window and navigation service
- page state lives in view models
- view markup stays mostly declarative
- future operational settings are grouped before the real backend connection exists

That gives later work a place to land.

## what broke

- **the domain is mostly absent.** the app builds, but the real operational workflow is not
  implemented.
- **starter labels remain.** generated app names, demo buttons, color blocks, and demo
  version text are useful as evidence of stage, but they should not be mistaken for product
  language.
- **machine/API fields are stubs.** connection test and machine-info actions
  simulate success rather than calling a real service.
- **nullable warnings remain.** configuration properties, navigation fields, color lists,
  and base-path setup need sharper initialization.
- **local tool metadata is not part of the archive.** assistant/project-management config
  in the source is operational context, not public article material.

## what i understand now

This kind of shell is easy to underrate because it does not yet "do the thing." But the
first frame of a desktop operations app matters. If the host, navigation, settings model,
and page boundaries are wrong, every later screen pays for it.

The right label is still important. This is not a finished console. It is a frame for one:
enough structure to compile, navigate, theme, and hold future settings; not enough
behavior to claim field use.
