# build_log_003 // finished_batch_packout_report_export

## what this is

A Windows-first reporting tool for production batches. It reads completed batch data from a
machine database, lets an operator review or configure the report, and exports packout
summaries as PDF and CSV. The production shape is a desktop app paired with a local
background service; an older script/CLI path remains as a reference while the system moves
toward the installed app.

The important part is the unattended loop: when a batch is finished, wait a short grace
period, render the configured report formats, write the files, and remember what happened.

<TerminalBlock title="export loop (sanitized)" prompt="">
{`finished batch
  -> grace window
  -> active report definitions
  -> render pdf / csv / html
  -> write output file
  -> record ledger state`}
</TerminalBlock>

## why it exists

Batch reports are not valuable because they exist in a database. They become valuable when
someone can hand them to the next workflow without rebuilding the facts by memory. Before
this build, the fragile part was the handoff: choosing the right batch, matching weights to
the right buckets, producing the right export, and not accidentally repeating work.

The goal was narrow: make a finished batch produce a durable report artifact, repeatedly,
on a Windows machine, without requiring the operator to babysit every export.

## what changed

The system evolved from a script-driven reporting tool into a local production app:

- a desktop UI for choosing batches, configuring sources, and running reports
- a same-machine background service for rendering and auto-export
- a local config store for data sources, report settings, grade rules, and export history
- a self-contained installer so the target machine does not need a runtime already installed
- tests around export idempotency, retry timing, PDF validity, weight units, and summary layout

That shape matters because the report is only one slice of the problem. The harder system is
installation, configuration, reruns, failed exports, and the small mismatches between what
the machine stores and what the operator expects to read.

## the ledger was the quiet design choice

The export worker does not just render files. It asks whether this source, batch, report,
and format already succeeded and whether the file still exists. If the previous run failed,
it waits until a retry is due. If work starts, it writes a running state before rendering,
then records success or failure after the renderer returns.

That is the difference between a scheduled task and an operating loop. A scheduled task only
fires. A ledgered loop remembers.

## what worked

The useful pieces are boring in the right way:

- finished batches are filtered before export, so open batches do not produce premature
  reports
- output filenames are generated from stable report and batch metadata
- report formats are explicit, so PDF/CSV/HTML can be enabled without changing the worker
- PDF output is checked for a real PDF header before it is treated as successful
- failed renders get a next-retry time instead of hammering the same broken path every run

None of that is glamorous. It is the stuff that keeps an operator from having to remember
which button was already pressed.

## what broke

- **the vocabulary fought back.** machine categories, weight buckets, and customer-facing
  labels did not line up cleanly. the report had to become a translation layer, not just a
  table dump.
- **units matter late and everywhere.** once kilograms and pounds both exist, every summary
  row, export, and test has to prove which unit it is showing.
- **a local service is still a service.** keeping the desktop app and service on the same
  machine avoids a public server, but it adds startup, install, health-check, and
  credential-storage concerns.
- **finished means the source says finished.** auto-export depends on the machine database
  marking a batch complete. if that completion signal is missing or delayed, the loop waits.
- **the old path remains useful but confusing.** keeping the script version as a parity
  reference helps migration, but it also means documentation must keep saying which path is
  production and which path is legacy.

## what i understand now

The real artifact is not the PDF. The real artifact is the guarantee that a completed batch
can become a report exactly once, be retried when the render fails, and be recreated only
when there is a reason. The report renderer is just the visible end of that contract.

The pattern generalizes: if a task bridges machine data and operator paperwork, start with
idempotency and evidence of completion. The UI can improve later. The ledger has to be right
from the beginning.
